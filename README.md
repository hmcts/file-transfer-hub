# File Transfer Hub

This repository contains the Terraform and application code for the FTPS-based file transfer service deployed on Azure Container Apps.

## Deployment Notes

- `components/core` manages the shared Azure infrastructure, including networking, Log Analytics, Key Vault, and the temporary nonprod SFTP test target.
- `components/container-app` manages the Azure Container Apps runtime for the FTPS service and resolves its shared Log Analytics workspace and Container Apps subnet from the deployed `core` resources.
- `app` contains the ProFTPD-based FTPS container and the `lftp` forwarding logic.

## Key Vault Secrets

The FTPS runtime reads its login details and certificates from Key Vault.

For staff managing FTPS users in the Azure Portal, the main secret to update is:

- `ftps-local-users`: preferred secret for FTPS logins

This secret contains a JSON array of username and password pairs. Each object becomes a valid FTPS login for the service.

Example `ftps-local-users` value:

```json
[
  {
    "username": "ftpssvc",
    "password": "replace-me"
  },
  {
    "username": "partner-a",
    "password": "replace-me-too"
  }
]
```

### Managing FTPS Users In Azure Portal

1. Open the Azure Portal.
2. Open the Key Vault used by the target environment.
3. Go to `Secrets`.
4. Open the `ftps-local-users` secret.
5. Create a new secret version with the full updated JSON payload.
6. Add, remove, or change users by editing the JSON array.
7. Save the new version, then run the normal deployment pipeline so the Container App picks up the updated secret value.

Rules for editing `ftps-local-users`:

- The value must be valid JSON.
- The top-level value must be an array.
- Every entry must include non-empty `username` and `password` fields.
- Usernames must be unique within the array.
- When changing users, replace the whole JSON array with the new complete list.

The FTPS runtime also expects these Key Vault secrets to exist:

- `ftps-local-username`: legacy single-user login secret kept for backward compatibility
- `ftps-local-password`: legacy single-user password secret kept for backward compatibility
- `ftps-storage-sftp-username`: username used by the service to forward files to the downstream SFTP target
- `ftps-storage-sftp-password`: password used by the service to forward files to the downstream SFTP target
- `ftps-certificate-pem`: FTPS server certificate in PEM format
- `ftps-certificate-key-pem`: FTPS server private key in PEM format

Staff using the Azure Portal should normally manage FTPS users through `ftps-local-users` rather than editing the legacy `ftps-local-username` and `ftps-local-password` secrets.

## Environment Behavior

- Nonprod: Terraform generates the FTPS login secrets, including the combined `ftps-local-users` JSON secret, and the temporary SFTP forwarding secrets. The FTPS certificate must still be supplied from Key Vault rather than generated by Terraform.
- Prod: Terraform does not generate the FTPS login or certificate secrets. They must already exist in Key Vault before the container-app deployment runs.

### Which Key Vault To Use

- Nonprod: use the Key Vault created by the `core` deployment for the nonprod environment.
- Prod: use the Key Vault created by the `core` deployment for the prod environment, unless the certificate is intentionally stored in a different vault.

To use a certificate from a different Key Vault (for example, Acmebot), set `ftps.certificate_key_vault_id` and point `ftps.certificate_secret_name` and `ftps.certificate_key_secret_name` to PEM-format secrets in that vault.

## Current Nonprod Test Approach

Until the real downstream SFTP server is available, nonprod uses the project storage account as a temporary SFTP forwarding target.
